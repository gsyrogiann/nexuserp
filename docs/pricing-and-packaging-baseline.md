# NexusERP Pricing and Packaging Baseline

## Purpose

Capture a conservative first-pass commercial structure so pilot and early sales
conversations remain consistent.

## Current Recommendation

Use a staged commercial motion:

* controlled pilot
* paid early adopter
* full GA package later

## Packaging Baseline

### 1. Pilot Package

Use for:

* selected design partners
* customers helping validate workflows
* tightly scoped onboarding

Should include:

* limited user scope
* guided setup
* weekly feedback loop
* explicit known limitations
* no broad feature promises outside written scope

### 2. Early Adopter Package

Use for:

* first paying customers after pilot validation
* customers whose scope matches the stable core flows

Should include:

* controlled production use
* support expectations in writing
* documented release and known-issues policy
* clear boundaries for integrations and experimental features

### 3. Future GA Package

Only promote once:

* pilot evidence exists
* support process is stable
* legal/commercial docs are finalized
* release process is operationally repeatable

## Pricing Inputs That Must Be Decided

Before pricing is finalized, decide:

* SaaS / on-prem / hybrid
* per-account vs per-user pricing
* whether Gmail/VoIP/AI are included or optional add-ons
* whether onboarding is bundled or paid separately
* what support response expectations are included by package

## Commercial Guardrails

Do not promise:

* unlimited custom work inside a standard package
* unbounded support for pilot pricing
* production guarantees not supported by the operational baseline
* integration maturity beyond what is currently validated

## Recommended Near-Term Motion

For the next phase, use:

* free or discounted pilot for high-learning accounts
* paid onboarding for customers needing migration/setup effort
* tightly scoped early adopter pricing before broad GA messaging
